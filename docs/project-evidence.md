# Project evidence and confidentiality boundaries

This portfolio uses concise public case studies, not a complete engineering archive. Published statements must be supportable by repository code, tests, existing project records or owner-approved public documents. If implementation evidence is incomplete, the copy stays general and planned work remains clearly labelled as future work.

## Portfolio website

The portfolio case study is directly supported by this repository:

- React components and Vite configuration support the frontend architecture.
- `functions/api/contact.js` and its tests support the Pages Function, validation, Turnstile and Resend descriptions.
- `.github/workflows/validate.yml` supports the CI description.

- the recorded Lighthouse scores remain explicitly described as local laboratory results, not field data.

## Generic Serial Product Test Framework

The retained project record supports a Python command-line workflow with serial-device discovery, shared and product-specific test logic, explicit pass/fail presentation and CSV result logging. The public case study does not claim a PyQt interface, Excel logging or completed queue/batch execution.

The framework source is not published in this portfolio repository, so the case study avoids implementation metrics and detailed internals that cannot be rechecked here. Batch execution, richer reports, configurable plans and stronger recovery remain future improvements.

## Jetson Nano Camera Benchmarking System

The retained project record supports Python experiments using OpenCV and GStreamer on Jetson Nano, including comparisons of runtime, CPU load, output file size and hardware-assisted H.264 encoding. It documents trade-offs rather than declaring one universally best pipeline.

No raw benchmark dataset or reproducible results table is present in this repository. The case study therefore publishes no unverified numeric performance claims and keeps automated runs, clearer exports and wider resolution/frame-rate coverage as future improvements.

## Experience evidence

The experience section uses only owner-approved facts already present in the public resume: role, organisation, period, high-level sensor-board work, a Google Apps Script workflow and technical documentation. It does not publish personal contact details, customer information, internal identifiers, proprietary screenshots or confidential biomedical work.

## Review checklist for future edits

Before expanding a case study:

1. Identify a non-confidential source that supports the statement.
2. Separate completed work from experiments and future improvements.
3. Avoid precise metrics unless the method and result can be verified.
4. Remove customer, dataset, device and internal workflow details that are not intended for publication.
5. Use British English and describe individual contribution without exaggeration.
